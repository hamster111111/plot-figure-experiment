对 F-A 方向的关键洞察：

① 91.7% 的失败来自"答案不匹配"，本质是数据推理/计算能力不足，不是数据编造。模型看到了正确的源数据但算错了。

② 真正的"数据保真"问题（图里画了源数据中不存在的数值）集中在 chart correctness < 3.5 的 74 个样本（6.4%），比 MatPlotBench 上的比例更高。

③ 15.7% 的"格式差异"（百分比 vs 小数）暴露了评测方法的缺陷——这是 F-A 可以改进评测维度的切入点。

④ 代码执行率 98.8% + 可读性 4.75/5 说明画图本身不是瓶颈，"理解数据 → 正确推理 → 忠实呈现"才是真正的 gap。

对 F-A 方向的关键洞察：

① 953 个 data_check 失败是"数据保真度"问题的直接证据。VisEval 的 legality check 精确验证图里画的数据是否来自源表——DeepSeek-V3 在这个维度上大量失败。这不是"推理算错"（Text2Vis 的问题），而是"数据映射错"——模型选错了列、聚合方式不对、或者画了不存在的数据。

② 和 Text2Vis 形成互补：Text2Vis 上 91.7% 失败来自答案推理错误，VisEval 上 70%+ 失败来自数据映射错误。两个 bench 从不同角度证明了"数据保真"是真实且严重的问题。

③ VisEval 的 data_check 天然适合做 F-A 的评测维度——它已经在验证"图里画的数据是否来自源表"，和 NumericTrace 的思路高度吻合。

METAL — 多 Agent 迭代图表生成（ACL 2025）
基本信息

论文：METAL: A Multi-Agent Framework for Chart Generation with Test-Time Refinement

会议：ACL 2025 CCF-A

代码：GitHub — 已 clone 到 D:\Code\autoReaserch\METAL\

Benchmark：ChartMimic（chart-to-code mimicking，给参考图生成代码复现）

架构

三 Agent 迭代循环：

  • Generation Agent — 根据参考图 + 指令生成 matplotlib 代码
  • Critique Agent — VLM 对比生成图 vs 参考图，给出修改建议
  • Revision Agent — 根据 critique 修改代码，循环直到满意或达到最大轮次

本质是 Generate → Critique → Revise 的 test-time refinement loop。

与 F-A 的关系：METAL 做的是 chart mimicking（给图复现），不是 NL→chart 或 data→chart。它不涉及数据保真度问题——输入是参考图片而非源数据。和 F-A GroundedChart 方向完全正交。但其 multi-agent critique loop 的架构思路可以借鉴。
RAGvis — KG 驱动的 EDA 可视化推荐（EMNLP 2025）
基本信息

论文：RAGvis: Retrieval-Augmented Generation for Automatic Visualization Recommendation

会议：EMNLP 2025 CCF-B（Google Research）

代码：GitHub — 已 clone 到 D:\Code\autoReaserch\RAGvis\

基础设施：PostgreSQL + PGVector（列 embedding 存储，25K 条）、Eclipse RDF4J（知识图谱，2M triples）、fasttext cc.en.300.bin（7GB 列 embedding 模型）

架构

离线阶段：从 Kaggle 10 万+ notebook 中提取 EDA 操作，构建知识图谱（列→EDA 操作→图表类型）+ 列 embedding 向量库

在线阶段：

  • Data Profiling — 对输入 CSV 的每列做类型检测 + embedding 生成（需 fasttext 50d + 300d）
  • KG Retrieval — 在 PG 中找相似列 → 在 RDF4J 中查这些列的历史 EDA 操作
  • LLM Refinement — 用 LLM 从检索到的 EDA ops 中筛选最有价值的
  • Code Generation — LLM 生成 Python 可视化代码
  • Execute & Fix — 执行代码，失败则 LLM 自动修复（最多 3 轮）
实测结果（在 VisEval 失败案例上）

测试案例：vis_id=1018，NL query="A pie chart showing the number of customer addresses in each premises type"

RAGvis 返回的 EDA ops：

  • bar chart on address_type_code
  • distribution on date_address_from
  • distribution on date_address_to
  • scatter on (date_address_from, date_address_to)
  • heatmap

结果：5 个代码片段全部执行失败，0 个图表生成成功。

根本问题：RAGvis 不接受 NL query 作为输入。它是 EDA 推荐系统（给 CSV → 推荐可视化），不是 NL→chart 生成器。对于"画一个 pie chart"这种精确需求，它返回的是 bar/distribution/scatter/heatmap — 完全不对口。

论文声称的 ~100% Pass Rate 解读

RAGvis 论文在 VisEval 上报告了接近 100% 的 "pass rate"，但这个指标只衡量代码可执行性（executability），不衡量数据正确性。

  • 不测 data_check — 不验证图里画的数据是否来自源表
  • 不测 NL 对齐 — 不验证生成的图是否匹配用户的自然语言描述
  • 只测 executability — 代码能跑起来就算 pass

这和 VisEval 官方的 Legality Check（chart_type + data_check + order_check）是完全不同的评测维度。

搭建踩坑记录
  • 内存杀手 fasttext cc.en.300.bin 加载需 ~7GB RAM，cc.en.50.bin 需 ~1.2GB。16GB 机器跑不动原版，需用 sentence-transformers (~80MB) 替代
  • SPARQL 慢 RDF4J 查 2M triples，单次 fetch_eda_operations 需 ~60s。5 列 × 2 similar = 10 次查询 ≈ 10 分钟
  • Windows 兼容 fasttext 在 Windows + Python 3.12/3.13 编译失败，需用 fasttext-wheel 替代
  • Docker 依赖 需要 PG + RDF4J 两个容器，KG 数据 ~2M triples 需从 GitHub Release 下载
对 F-A 方向的关键洞察：

① RAGvis 和 F-A GroundedChart 解决的是完全不同的问题。RAGvis 做"给数据推荐 EDA"，F-A 做"根据 NL 描述生成数据保真的图表"。

② RAGvis 的 ~100% pass rate 是 executability metric，不是 data fidelity。在 VisEval 的 data_check 维度上，RAGvis 的架构根本不适用。

③ 这对 F-A 是好消息：即使是 Google Research 的 EMNLP 2025 工作，也没有触及"数据保真度"这个维度。我们的方向有明确的差异化价值。

④ RAGvis 的 KG 检索思路（从历史 EDA 中学习）可以作为 related work 讨论，但不构成直接竞争。
