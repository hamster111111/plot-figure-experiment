ClawEvalKit — Thought Page

这页记录的是“我是怎么确认它的 git workflow 的”，不是结论复述。

First Interpretation

第一反应是：这可能只是一个普通仓库，里面放了一些文档 HTML，然后用 GitHub Pages 从 docs/ 或 gh-pages 分支发布。

因为你提到“维护一个 io，把 web 里面的 html 都放过去”，这在经验上常常对应 user pages repo 或单独 gh-pages 分支。

Discriminating Questions

最关键的判断问题只有三个：

  • 有没有 .github/workflows，以及触发条件是什么
  • 部署源目录是 docs/、work/，还是某个分支
  • 当前 main 上的 workflow 是否自洽，还是历史残留
Assumptions Revised

我最初确实怀疑它会是 docs/ 发布流，因为仓库里存在 docs/，而且有 dashboard HTML。

但看完 deploy.yml 后这个假设被推翻：workflow 明确上传的是 work/，不是 docs/。

接着又发现当前仓库没有 scripts/gen_index.py 和 work/，这说明不能只看当前树，还必须追 commit 历史。

Strongest Interpretation

最终最强解释是：

  • ClawEvalKit 历史上有过一套完整的静态 HTML 发布链路
  • 那套链路的核心是 main push → 生成索引页 → 上传 `work/` → GitHub Pages 发布
  • 后面有人清掉了站点产物，但没有把 workflow 一起维护到一致状态

所以这个 repo 能提供的价值是“发布模式参考”，而不是“当前 main 可直接复用”。

What Remains Uncertain

仍有两个不确定点：

  • 仓库设置页里 GitHub Pages 最终绑定的是 project pages 还是 custom domain，这里没直接看到
  • 你师兄说的“维护一个 io”是单独的 username.github.io 仓库，还是泛指要做一个 Pages 站点

这两个差异会影响最终目录结构，但不会改变上面的核心判断。

Practical Consequence

对你最直接的启发是：

  • 不要照抄当前 main
  • 要么复刻它历史上的 work/ + 自动索引 + deploy-pages 方案
  • 要么更干脆地把你自己的 web/ 作为唯一内容源，重新写一个更干净的 workflow
