ClawEvalKit — Git Workflow 与 Pages 部署分析

Repository approved and inspected on 2026-04-22 · 只聚焦 git / GitHub Pages / HTML 发布路径

一句话结论

ClawEvalKit 采用的是“main 分支 push 触发 GitHub Pages 部署”的 workflow，不是 gh-pages 分支流；但当前 main 上的 Pages workflow 处于失配状态。

核心原因：workflow 还在，但它依赖的 scripts/gen_index.py 和 work/ 目录已经被后续提交清掉。

事实：当前 workflow 是什么
`.github/workflows/deploy.yml` 的实际逻辑

触发条件是 push 到 main 或手动触发，不依赖 gh-pages 分支。

流程是：checkout → 运行 python3 scripts/gen_index.py → 把 work/ 作为 Pages artifact 上传 → deploy-pages 发布。

证据：deploy.yml

on: push: branches: [main] ... - name: Generate index.html run: python3 scripts/gen_index.py ... with: path: 'work'
分支层面

远端只看到 main 和一个修复分支 fix/large-files，没有公开的 gh-pages 分支。

这说明它走的是 GitHub Actions Pages artifact deploy，不是“把静态文件直接推到 gh-pages 分支”。

事实：为什么说当前 main 上是断的
项目当前状态含义
.github/workflows/deploy.yml存在Pages 部署入口还在
scripts/gen_index.py当前 main 不存在workflow 第一步会失败
work/当前 main 不存在即使脚本不报错，artifact 目录也缺失
docs/存在，但未被 workflow 使用不是当前 Pages 部署源目录
关键判断：它不是“docs 发布流”，而是历史上曾经存在的 work/ 站点发布流被删了一半，只剩 workflow 壳子。

这对你现在的任务意味着什么
如果你师兄说“照这个 workflow 来”

真正可复用的模式不是某个现成脚本，而是这三个约束：

  • 内容源目录应统一，历史上是 work/
  • 入口页由脚本自动扫描生成，而不是手写
  • 部署由 main push 触发，GitHub Actions 直接发 Pages artifact
如果你的目标是“把本地 `web/` 里的 HTML 都放过去”

最稳妥的迁移方式不是照搬它现在的 main，而是借用它历史上完整的思路：

  • 把你现在的 web/ 作为静态内容源，或者同步到目标 repo 的 work/
  • 补一个自动生成 index.html 的脚本
  • 保留 push main → deploy-pages 的 workflow

换句话说：要学的是发布模式，不是照抄当前 broken 的目录状态。

建议的执行方案
方案做法适合评价
A. 复刻 ClawEvalKit 历史方案目标 repo 用 work/，写扫描脚本，Actions 部署你师兄想延续那套命名最贴近原始意图
B. 简化成 `web/` 直发workflow 直接上传 web/，不改目录名你只想稳定发布现有页面工程上更干净
C. `docs/` 方案把 HTML 搬到 docs/ 再发 Pages想兼容传统 GitHub Pages 习惯和 ClawEvalKit 当前 workflow 不一致
推荐：如果没有强制历史包袱，优先选 B。如果你师兄明确要“按 ClawEvalKit 那个 workflow”，就选 A，但要把缺失的脚本和目录一并补回来。
