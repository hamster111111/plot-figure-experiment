差异化论证 — 为什么 F-A 方向成立（以及现在该怎么定义）

一句话：数据保真度是科学图表生成中一个真实、可定位、且无人解决的问题。但我们的证据强度在不同 benchmark 上差异很大——VisEval 是强支撑，Text2Vis 需要谨慎解读，MatPlotBench 基本不相关。
F-A 现在最好变成什么？
推荐版本

F-A = faithful language-to-chart generation 的 execution-grounded verification-and-repair framework

  • 输入：NL query + structured data + 初始 plotting code
  • 验证对象：NL 意图对应的数据操作是否真的体现在代码和最终图表数据里
  • 修复对象：group-by / filter / join / aggregation / sort / type 等操作级错误
  • 输出目标：不是只生成能跑的代码，而是生成 data-faithful 的图表代码
不推荐版本
  • 不要再讲成泛化的"科学图表真实性"或"图表防幻觉"系统，范围太大
  • 不要把它写成单纯的 data check agent，这样 novelty 太弱
  • 不要把 Text2Vis 的低分直接拿来证明必要性，论证会失焦
  • 不要把视觉质量、答案抽取、示意图生成混进同一个故事里
最稳的 EMNLP 风格 claim：我们研究的是 语言意图 → 数据操作 → 图表数据 三者之间的语义一致性，并提出一个可执行、可归因、可修复的验证框架来提升图表生成的数据保真度。
Plot/Figure 的数据保真度为什么没人做？

不是因为太简单或没必要。恰恰相反，原因是：

  • 领域 framing 问题：大家把任务定义成了"代码生成"或"视觉还原"，评测指标也跟着走——MatPlotBench 评视觉相似度，ChartMimic 评图像匹配。没人设计指标去量化"图里的数据对不对"
  • 沉默错误：数据保真度错误是"沉默"的。图表看起来完全正常（Text2Vis 里 readability 5.0, correctness 5.0 的案例，答案照样错），不像执行失败那样显眼
  • 评测空白：VisEval 是少数做了 data_check 的 benchmark，而且比较新。在它之前，这个维度根本没有被系统性地测量过

结论：不是没必要，是没人意识到这是个独立的问题维度。

Benchmark 证据强度 — 诚实评估

三个 benchmark 对 F-A 的支撑力差异很大，不能一概而论。

VisEval — 强支撑

953 例 data_check 失败中：

  • missing_groupby (551) + missing_filter (107) + wrong_agg (27) = 685 例，占 72%
  • 这些是货真价实的数据保真度问题——图里画的数据就是错的
  • F-A 直接对口，根因明确，可针对性解决

口径转换实证：去掉 data_check 后 pass rate 从 36.8% 跳到 85.2%（+48.4pp）。同口径下我们与 GPT-4 baseline (75.27%) 和 Gemini-2.5-Pro (83.71%) 处于同一水平——说明模型代码生成能力没问题，瓶颈就在数据保真度。

结论：VisEval 是 F-A 最强的论据。口径转换数据让这个论据从"定性判断"变成了"定量证明"。

Text2Vis — 需要谨慎解读 ⚠️

1,062 例 answer_mismatch 中，大部分不是图表数据错误：

  • wrong_entity (544): 很多案例图表本身是对的（readability 5.0, correctness 5.0），错在"从图中提取答案"这一步。这是推理/阅读理解错误，不是图表数据错误
  • wrong_value (351): 同理，可能是计算推理错误，图表数据本身可能没问题
  • format_mismatch (92) + pct_vs_decimal (42) + partial_overlap (29) = 163 例：格式/评测标准问题，根本不是真正的错误
  • exec_failure (172) + low_correctness (75) = 247 例：这部分 F-A 可能有帮助

结论：Text2Vis 的 33.4% pass rate 听起来很惨，但 F-A 能直接解决的可能只有 ~19% 的失败。不能把 Text2Vis 的数字直接拿来撑场面。

MatPlotBench — 弱支撑

评的是视觉相似度，不是数据正确性。

  • 43% 低分案例中，大部分是颜色、布局、样式差异
  • 只有 7 例 0 分（完全错误主题）可能涉及数据保真度
  • MatPlotBench 的评测维度和 F-A 的目标基本不重叠

结论：MatPlotBench 不适合作为 F-A 的核心论据。

失败的本质 — 三个 Benchmark 在说不同的事
Benchmark主因本质F-A 相关度
VisEval缺少 GROUP BY 聚合 (60%)代码级数据处理错误直接对口
Text2Vis答案提取/推理错误 (84%)模型理解力问题帮不上太多
MatPlotBench视觉样式差异渲染问题不对口

核心矛盾：VisEval 的失败是"数据就画错了"，Text2Vis 的失败更多是"数据画对了但答案算错了"。这两个问题需要不同的解决方案。

实验结果 — 有利 vs 不利
对我们有利
  • VisEval 953 例失败中 72% 是数据保真度问题，根因明确（missing_groupby 60%）
  • 口径转换铁证：去掉 data_check 后 pass rate 36.8%→85.2%，同口径下与 Gemini-2.5-Pro (83.71%) 持平——瓶颈就在数据保真度
  • 12 个竞品系统 0 个做数据保真度，空白是真实的
  • CoDA/VisPath/PlotGen 都没开源，无法直接对比，给了我们空间
  • 问题可复现、可量化、根因可定位——不是玄学
  • Plot/Figure 赛道和 Sci Illustration 赛道的区分清晰，我们的定位明确
对我们不利
  • Text2Vis 的失败大部分是推理/提取错误，不是图表数据错误。如果把 33.4% 当核心论据，站不稳
  • format_mismatch + pct_vs_decimal (134 例) 是评测标准问题，拿来凑数会被质疑
  • MatPlotBench 评的是视觉相似度，和数据保真度关系不大
  • missing_groupby 虽然占 60%，但修复方式相对直接，需要论证为什么需要一个系统而不是一条规则
  • 我们用的是 DeepSeek-V3（弱于 GPT-4/Gemini），reviewer 可能说"换个好模型就行了"
修正后的三个论据
论据 1: 问题真实存在（但范围比表面窄）
  • VisEval: 953 例 data_check 失败中 72% 是数据保真度问题；口径转换后 pass rate 从 36.8%→85.2%，48.4pp 的落差全部来自 data_check ← 强证据
  • Text2Vis: 大部分失败是推理/提取错误，F-A 直接相关的约 19% ← 需谨慎引用
  • 诚实表述：数据保真度问题在 NL+数据→图表 任务中真实存在且严重，但不是所有图表生成失败的主因
论据 2: 根因可定位（但需要超越简单规则）
  • 60% missing_groupby — 明确的代码级根因
  • 12% missing_filter — 可检测的逻辑错误
  • 风险：reviewer 会问"加个 GROUP BY 检查器不就行了？"
  • 应对：需要证明 F-A 是通用的数据验证框架，不只是规则匹配
论据 3: 竞品空白（真实且已验证）
  • Plot/Figure 赛道 7 个系统，0 个做数据保真度
  • Sci Illustration 赛道 5 个系统，与 F-A 完全正交
  • CoDA/VisPath/PlotGen 未开源（已验证），无法直接对比
  • RAGvis 实测 5/5 失败，AutoFigure 不验证数据
方向判断 — 值得做，但需要调整 framing

做的理由：

  • VisEval 的 60% missing_groupby 是一个明确的、可定位的、系统性的缺陷，且没有任何竞品在解决
  • 这个问题在实际应用中很危险——用户看到一张"看起来正确"的图表，但数据是错的
  • 竞品空白是真实的（12 个系统 0 个做数据保真度）

风险与应对：

  • 风险：Text2Vis 的数字不能直接用 → 应对：论文聚焦 VisEval 类任务，Text2Vis 作为补充而非核心
  • 风险：missing_groupby 看起来太简单 → 应对：证明 F-A 是通用框架（覆盖 groupby/filter/agg/type 等多类错误）
  • 风险："换个好模型就行了" → 应对：需要在 GPT-4/Claude 上也跑一遍，证明问题不只是弱模型的问题
F-A 最好长成一个什么系统？
  • Step 1. Intent / operator plan extraction — 从 NL query 中显式抽取维度、度量、聚合、过滤、排序、图表类型等结构化意图
  • Step 2. Execution-grounded verification — 执行代码或中间查询，拿到实际被画出来的数据，再与 operator plan / source data 对比
  • Step 3. Localized repair — 将错误定位到 missing_groupby / wrong_filter / wrong_agg / wrong_join / wrong_type 等操作级，再定向修复代码
  • Step 4. Final rendering sanity check — 确认修复后代码仍可执行，且不明显损害视觉可读性

关键点：这不是松散的"三个方向"，而是一个完整的 verify-and-repair pipeline。这样才能把问题讲成一个框架，而不是若干小技巧。

按 EMNLP 模板重写后的方向
1. Problem / Scene

目标场景：用户给出自然语言请求和结构化数据，希望模型直接生成可发表级别的图表代码。

  • 典型请求："show total sales by product category in ascending order" 这类请求同时包含维度、度量、聚合、过滤、排序和图表类型约束
  • 真实风险：模型往往能生成可执行、视觉上合理的图，但图中数据可能并不对应用户想要的 aggregation / filter / join 结果
  • 关键难点：这类错误是 silent failure。图表看起来正常，却在数据语义上是错的
  • 论文问题定义：faithful language-to-chart generation 不是只让代码能跑，而是让最终图表数据忠实对应自然语言意图和源数据操作
2. Limitation of Prior Work
  • 现有方法的优化目标错位：大多优化 executability、visual similarity、或 end-to-end answer quality，而不是数据操作语义本身
  • 缺少显式中间表示：现有系统通常不会把 query 中的 group-by / filter / join / aggregation 显式抽出来再验证
  • 缺少执行后验证：它们很少回到 source data 和实际被画出来的数据上，检查二者是否一致
  • 现有高分不等于高保真：强模型和强 agent 可以提高 pass rate，但在缺少 strict data_check 的口径下，高分并不能证明图表数据正确
  • 因此 reviewer-safe 的问题不是“模型还不会画图”，而是“现有系统没有验证 language intent 到 plotted data 的语义一致性”
3. Our Framework

推荐表述：We propose an execution-grounded verification-and-repair framework for faithful language-to-chart generation.

  • 核心思想：先把自然语言中的数据操作意图结构化，再把代码真实执行出来的数据 trace 抽出来，两者做语义对齐
  • 不是做一个泛化的“图表防幻觉 agent”：而是针对 structured data-to-chart generation 的 operator-level verification
  • 不是规则补丁：框架不只查 missing_groupby，而是统一处理 aggregation、filter、join、sort、type 等一组可组合的数据操作错误
  • 论文贡献点：把“意图表示 + 执行验证 + 局部修复”连成一个完整闭环，而不是单点 heuristic
4. Key Modules
ModuleFunctionWhy it matters
Intent Parser从 NL query 抽取维度、度量、聚合、过滤、排序、图表类型没有显式 operator plan，就无法判断代码到底偏离了哪里
Execution Tracer执行初始代码或中间查询，提取实际被绘制的数据表征把“模型打算做什么”变成“模型实际上画了什么”
Operator-level Verifier比较 operator plan、source data、plotted data 之间的一致性把错误定位到 groupby/filter/agg/join/type 等操作级，而不是只给一个总分
Localized Repairer基于错误类型定向修复代码，再重新验证把问题做成可修复的 framework，而不是一次性打回重写
Rendering Sanity Check确认修复后代码仍可执行且可读性不明显下降防止 fidelity 提升是以破坏可执行性或图表质量为代价
5. Results

当前已有证据：

  • VisEval 中 953 个 data_check failure 说明问题真实存在，而且 72% 可归因到 operator-level data processing errors
  • 36.8% → 85.2% 的口径拆解说明：原始代码生成能力不是唯一瓶颈，strict data fidelity 才是主要难点
  • Text2Vis 和 MatPlotBench 只能作为迁移或 sanity evidence，不能冒充主证据

如果要达到 EMNLP 主会水准，paper 至少要证明：

  • 主实验：在 VisEval 的 fidelity-sensitive 指标上取得显著绝对提升，最好达到 double-digit gain
  • 机制证据：提升不能只来自 missing_groupby，而要覆盖 filter / aggregation / join / type 等多个 error families
  • 泛化证据：除 VisEval 外，还要在第二个 benchmark 或高质量 operator diagnostic set 上证明收益
  • 强模型证据：至少在一个更强 backbone 上仍有增益，否则 reviewer 会归因于 base model 太弱
  • 代价控制：executability 和 visual quality 不能明显退化，否则会被认为只是 trade-off 而非真实进步
严格判断：如果最后只能证明“修一个高频 groupby bug”，或者只能在单一 benchmark 上提升，这更像 Findings / ARR / workshop。只有当它被证明是一个跨 operator、跨模型、跨评测设置都成立的 verification-and-repair framework，才像 EMNLP main 的叙事。
给师兄的建议：F-A 值得继续做，但必须收窄成 faithful language-to-chart generation 的 verification-and-repair 问题，而不是泛化的"真实性"口号。主动承认 Text2Vis 和 MatPlotBench 不是核心证据，把 VisEval + operator-level repair 作为主线，会更像一篇可 defend 的 EMNLP 论文。下一步建议：先做一个最小 prototype，证明它不只会修 missing_groupby，而能稳定覆盖 groupby/filter/agg/type 等多类错误。
